# 11.1 共享内存式同步模式

> 程序的构造可以用更简单的基础原语来表达，这一事实是一个有力的保障：这些原语所包含的内容，
> 能与程序语言的其余部分在逻辑上保持一致。
> ——  C. A. R. Hoare

多数语言把互斥、信号量、条件变量当作并发的「一等公民」。Go 反其道而行：它把
[channel](../ch10chan) 与 select 这套消息通信式同步作为语言层面的核心，而把传统的互斥、原子、
条件变量、线程本地资源等，降格为标准库 `sync` 与 `sync/atomic` 里的**同步模式**。这一取舍呼应
了 Go 的并发箴言：「不要以共享内存的方式通信，而要以通信的方式共享内存。」

但「以通信共享内存」是倡导，不是禁令。有些问题，用一把锁保护一小段共享状态，比绕一圈用
channel 更直接、也更快。`sync` 包正是为这些场景准备的工具箱。本章逐一拆解它们的实现与取舍：

- **互斥锁**（[11.2](./mutex.md)）：最基本的临界区保护，看它如何在吞吐与公平间权衡。
- **原子操作**（[11.3](./atomic.md)）：最贴近硬件的一层，无锁编程的基石。
- **条件变量**（[11.4](./cond.md)）：等待某个条件成立，以及为何在 Go 里它常被 channel 取代。
- **同步组**（[11.5](./waitgroup.md)）：等待一组 goroutine 全部完成。
- **缓存池**（[11.6](./pool.md)）：复用临时对象，给 GC 减负。
- **并发安全散列表**（[11.7](./map.md)）：读多写少场景下的并发 map。
- **上下文**（[11.8](./context.md)）：在调用树间传播取消与截止时间。
- **内存一致模型**（[11.9](./mem.md)）：上述一切之所以成立的底层契约。

读完本章会发现一条贯穿始终的主线：每一个原语都是在**正确、简单、性能**三者之间的一次具体
取舍，而 Go 的偏好始终是把简单与正确放在性能之前。
