# 3.4 模块链接

编译器（[3.2](./compile.md)）把每个包变成一个目标文件，但目标文件还不能直接运行,它们彼此
引用着对方的函数与变量，还需要运行时的支撑。把这些碎片拼成一个完整、可执行的程序，是**链接器**
（`cmd/link`）的活。这一节看链接器做了什么，以及 Go 在链接上的几处独特选择。

## 3.4.1 链接器做的事

链接器接过所有目标文件，完成几件关键工作：**符号解析**,把每处对外部符号（别的包的函数、
全局变量）的引用，绑定到它真正的定义;**重定位**,在符号的最终内存地址确定后，修正所有引用处
的地址;**布局**,把代码、数据、只读数据等安排进可执行文件的各个段;以及**死代码消除**,
未被引用到的函数和变量不会被链接进最终二进制，以减小体积。最后产出一个可被操作系统加载执行的
文件。

关键的一点：**运行时也是被链接进来的**。你的 `main` 包、它依赖的所有包、以及整个 Go 运行时
（调度器、GC、内存分配器），都被链接器拼进同一个二进制。这正是 Go 程序"自带运行时"的由来,
没有外部的虚拟机或解释器，运行时就静静地待在你的可执行文件里。

## 3.4.2 静态链接：自带一切的二进制

Go 的一个标志性选择是**默认静态链接**。一个纯 Go 程序编译出的，通常是一个**不依赖外部共享库**
的、自包含的可执行文件,把它拷到另一台同架构同系统的机器上就能跑，无需安装任何运行库。这与
C/C++ 程序常见的"依赖一堆 `.so`/`.dll`、换台机器就缺库"形成鲜明对比，也是 Go 在容器与云原生
时代大受欢迎的原因之一,一个 `FROM scratch` 的极小镜像里塞一个 Go 二进制就能工作。

代价也有。静态链接让每个二进制都自带一份运行时与用到的库代码，**体积**比动态链接的大;一旦
某个库有安全更新，得**重新编译**而非替换共享库。涉及 cgo（调用 C 代码）时还会引入对 libc 的
动态依赖，破坏纯静态性。这又是一处典型取舍：Go 用更大的二进制、换取**部署的极致简单与
可移植**,在它瞄准的服务端/云场景里，这笔交易非常划算。

## 3.4.3 链接器的演化

Go 的链接器同样源自 Plan 9 传统（[2.1](../ch02asm/asm.md)），多年来持续被改造。一个值得记的
节点是 Go 1.15 前后对链接器的**大幅重写**：引入新的目标文件格式与符号处理方式，显著加快了
链接速度、降低了内存占用,对大型程序尤其明显。这再次呼应 Go 对**构建速度**的一贯重视
（[1.1](../ch01intro/history.md)）：链接是构建流水线的最后一站，它快不快直接影响每一次
"改代码、重新跑"的体验。从编译到链接，整条流水线的设计目标始终一致,让大规模 Go 工程的构建
保持飞快。下一站，我们看这个链接好的二进制被操作系统加载后，如何把自己"开机"
（[3.5](./boot.md)）。
