# 13.1 垃圾回收的基本想法

垃圾回收（GC）把程序员从手动 `free` 中解放出来，代价是运行时要自己判断"哪些内存还有用、
哪些可以收回"。Go 的 GC 以**低延迟**为首要目标,它宁可牺牲一点吞吐与内存，也要把"卡顿"
（stop-the-world 停顿）压到亚毫秒级。这一节建立 GC 的理论坐标：可达性、标记清扫、三色抽象，
以及 Go 在 GC 设计空间里的定位。后续各节是这套基本想法的展开。

## 13.1.1 可达性即存活

GC 判断存活的依据是**可达性**：从一组**根**（全局变量、各 goroutine 栈上的变量、寄存器）出发，
凡能沿指针到达的对象都算**存活**，到达不了的就是**垃圾**。这是一个保守而安全的近似,它不真正
知道对象未来会不会被用，只知道"够不到的肯定用不上"。可达性把"内存是否有用"这个语义问题，
转化成了一个图遍历问题：从根开始遍历对象图，标记所有能到达的。

## 13.1.2 标记—清扫

最经典的实现是**标记—清扫**（mark-sweep，McCarthy 1960）：**标记**阶段从根遍历、标记所有可达
对象;**清扫**阶段扫过堆，把未被标记的回收。Go 用的正是标记清扫，而非**复制式**（copying，
把存活对象搬到新空间）或**引用计数**（每个对象记被引用数）。这是一组取舍：标记清扫不移动对象
（指针稳定、对 cgo 友好），但会产生碎片;复制式无碎片但要搬移、且浪费一半空间;引用计数及时
但有循环引用问题与计数开销。Go 选标记清扫，并用尺寸类分配（[12.1](../ch12alloc/basic.md)）把
碎片问题控制在可接受范围。

## 13.1.3 三色抽象

要让标记能与用户程序**并发**进行（这是低延迟的关键），需要一个能描述"标记进行到一半"的状态的
抽象,这就是 Dijkstra 等人提出的**三色标记**：

```mermaid
flowchart LR
    W["白色<br/>尚未访问，回收候选"] -->|被灰色对象引用，置灰| G["灰色<br/>已访问，但其引用尚未扫描完"]
    G -->|自身引用全部扫描完毕，转黑| B["黑色<br/>已访问且引用已全部扫描，判定存活"]
    B -.->|根与黑色对象不再变白| DONE["标记结束：仍为白色者即垃圾，交清扫回收"]
```

**白色**=尚未访问（回收候选）;**灰色**=已访问但其引用还没扫完;**黑色**=已访问且引用已全扫
（确定存活）。标记从把根置灰开始，不断取一个灰色对象、扫描它的引用（把引用到的白色对象置灰）、
再把它自己转黑，直到没有灰色对象。此时仍是白色的就是垃圾。三色抽象的价值在于：它把"标记的
进度"显式化，从而能在用户程序**同时修改对象图**的情况下，定义出必须维持的**不变式**
（[13.2](./barrier.md)）,这是并发 GC 能正确工作的理论基石。

## 13.1.4 并发的代价：三色不变式

并发标记的危险在于：标记进行时，用户程序（mutator）还在改指针,它可能把一个白色对象挂到一个
已经变黑的对象下，而黑色对象不会再被扫描，于是这个白色对象会被**误回收**。要防止这种灾难，
必须维持**三色不变式**（[13.2](./barrier.md) 详述）：粗略说，就是"不允许黑色对象直接指向白色
对象"，或它的弱化版本。维持不变式靠的是**写屏障**,在指针写入时插一小段代码做必要的标记。
Go 的并发 GC，本质上就是"三色标记 + 写屏障 + 精心设计的步调"这三件事的合奏,本章接下来正是
逐一拆解它们：写屏障（[13.2](./barrier.md)）、步调（[13.3](./pacing.md)）、标记
（[13.4](./mark.md)）、清扫（[13.5](./sweep.md)）、终止（[13.6](./termination.md)）。
