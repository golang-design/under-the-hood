# 写作公约（重写版）

本文是《Go 语言原本》自 2026 年重写以来所遵循的写作公约。
它的目的是把本书的写作哲学固化为可执行的规则，使所有章节风格一致、深度可控。
撰写或修订任何章节前，请先阅读本文。

本轮重写对标 **Go 1.26**。所有与实现相关的论断都应以 Go 1.26 源码为准
（交叉核对的参考检出为 `release-branch.go1.26`），并在涉及版本差异时显式标注版本号。

## 一、写作哲学：设计先行，而非源码先行

本书的旧版本最大的问题是**源码分析过度**：大段粘贴运行时源码、逐行翻译注释、
随实现细节起舞。源码每个版本都在变，逐行讲解很快过时，且掩盖了真正"永生"的东西——
**设计原理与工程权衡**。重写遵循自顶向下的叙事顺序：

1. **问题（Problem）**——这个组件要解决什么本质问题？约束是什么（并发、局部性、
   内存、可移植性、兼容性）？
2. **设计（Design）**——为了解决该问题，采用了什么核心思想？背后的理论基础是什么？
3. **演进（Evolution）**——这个设计是如何一步步走到今天的？经历过哪些方案、为什么被
   推翻或修订？（见下文第三节）
4. **权衡（Trade-offs）**——当前实现牺牲了什么换取了什么？边界条件与已知缺陷在哪里？
5. **实现（Implementation）**——只在必要时，用最小的代码或图示落地到具体机制。

读者读完一节，应当能回答"**为什么这样设计**"，而不仅仅是"**代码怎么写的**"。

## 二、代码的准入规则（关键）

这是一本讲语言设计与工程实践的书，**纯散文太干、读者学不到东西**。要大胆地用代码把机制讲具体，
同时避开旧版"源码搬运"的老毛病。被禁止的是后者，不是代码本身。

**应当多用（让机制可触摸）：**

- ✅ **伪代码**讲算法骨架：CAS 重试循环、调度循环、`findRunnable` 的查找顺序、channel 收发的
  判定、mutex 的模式切换等，用十几行伪代码远比一段散文清楚。
- ✅ **精简的结构体速写**：只列与设计相关的几个字段，并加注释说明每个字段为何存在
  （如 `hchan` 的 buf/sendq/recvq/lock、mutex 的状态字位域、g/m/p 的关键字段）。
- ✅ **简短的惯用法示例**：展示用户该怎么用、容易怎么用错（如 `select` + `default`、
  `sync.Once`、`atomic.Int64` 替代裸 `atomic.AddInt64`）。
- ✅ 引用真实源码时给出**符号名 + 文件路径**（如 `runtime.schedule`，`runtime/proc.go`）
  供读者查阅，正文里放经过裁剪的版本。

**仍然禁止（旧版的失败模式）：**

- ❌ 整段粘贴真实源码函数体。
- ❌ 逐行翻译源码注释。
- ❌ 照搬完整的结构体定义（几十个字段），而非裁剪出设计相关的那几个。

判断标准：代码是否让读者**更快地理解了某个机制或设计决策**？是则留，否则删。
目标是"具体而经过裁剪"，而非"抽象而干瘪"，也非"照单全收"。

## 三、演进史是一项独立的研究工作

"设计如何演进"无法从当前源码树读出——它活在
**Go 提案（go.googlesource.com/proposal）、设计文档、release notes、issue 跟踪器、
邮件列表与会议演讲**里。撰写演进史时：

- 以**一手资料**为准（提案、设计文档、commit、作者文章），核实后再断言；
  把这些线索当作"待核实的线索"，而非"可直接粘贴的事实"。
- 用版本号锚定关键转折（如"Go 1.5 引入并发标记清除"、"Go 1.14 异步抢占"、
  "Go 1.19 内存模型修订"）。
- 解释**为什么改**，而不仅是**改了什么**。

## 四、图：以 Mermaid 为主

- 优先使用 **Mermaid**。**统一使用 ```mermaid 围栏代码块**，不要使用 `{{< mermaid >}}`
  shortcode。围栏代码块在 GitHub 上原生渲染，不会破坏直接阅读 `.md` 的体验；站点侧由
  `layouts/_default/_markup/render-codeblock-mermaid.html` 渲染钩子负责渲染（mermaid v11）。
  流程图、状态机、时序图、类图、时间线均可。
- 文字能讲清楚的，不要画图；图要承载文字不便表达的**结构、状态迁移、时序关系**。
- **图必须经实际渲染验证**，不能只看是否解析通过。例如 `gantt` 不适合表示「瞬时事件落在并行
  时间线上」这类图（里程碑会被挤到边缘、坐标轴退化），这类「多方时序」改用 `sequenceDiagram`
  （纵向自上而下即全局次序，箭头指向共享的「内存」参与者）。验证可用 `mmdc` 渲染成图后核对。
- 仅当 Mermaid 无法表达（如精确的内存布局、像素级示意）时，才使用静态图片，
  源文件放入 `book/assets/raw/`。
- 旧版散落的 `.png` 截图应在重写相应章节时尽量用 Mermaid 重绘。

Mermaid 用法示例：

```markdown
{{< mermaid >}}
stateDiagram-v2
    [*] --> Runnable
    Runnable --> Running: 被调度
    Running --> Runnable: 被抢占
    Running --> Waiting: 阻塞
    Waiting --> Runnable: 就绪
    Running --> [*]: 退出
{{< /mermaid >}}
```

## 五、理论深度

本书面向具备基本 CS 素养的读者，鼓励在恰当处补足理论：

- 给出**形式化定义**而非含糊描述（如 happens-before 偏序、DRF-SC、线性一致性、
  工作窃取的随机化分析）。
- 在引入概念时给出**最小自洽**的背景，使该节不必跳读即可理解。
- 把 Go 的具体取舍放到更大的**学术与工业脉络**中（CSP、内存模型、GC 算法谱系、
  调度理论）。
- **数学公式一律用行内 LaTeX**：行内用 `$...$`，行间用 `$$...$$`。GitHub 原生渲染，
  站点侧由 KaTeX（已 vendored 于 `themes/book/static/katex/`，并在 `config.toml` 的
  goldmark passthrough 中放行）渲染。不要用纯文本拼写 ∀、≤、⇒ 等符号。

## 六、学术与一手文献

- 凡有相关学术文献或经典论文，应在节末"延伸阅读"中给出**可核实的引用**
  （作者、标题、出处、年份，尽量附 URL）。
- 引用前核实可达性与元数据准确性，不要凭记忆杜撰。
- 一手资料（论文、提案、设计文档、作者博客）优先于二手转述。

## 七、面向网站的可视化（可选增强）

站点本身是一个交付载体。在有助于理解的地方，欢迎加入**交互式 / 动画可视化**
（如内存重排序、调度器工作窃取、GC 三色标记的动态演示）。

- 优先用轻量、自包含的方案（内联 `<script>` / `<canvas>` / CSS 动画 / 小型 Hugo
  shortcode），避免引入重型构建依赖。
- 动画必须有**对应的静态降级**（图或文字），保证离线 / 打印 / 不支持 JS 时内容完整。
- 可视化是锦上添花，不能成为理解某节的**必要条件**。

## 八、语言风格

**第一原则：沿用本书原有的作者声音，为读者而写。** 重写不是换一种腔调，而是在原作者的
语气里把内容更新、加深。原作者的声音是：温润、个人化、博雅而流畅，以「我们」与读者同行，
预判读者的疑问并主动作答（「读者可能好奇……」），需要表达己见时用「笔者」，善用「」『』
而非滥用加粗，章节常以一句耐记的话收尾。写每一段都要想着读者此刻在想什么、卡在哪里，
而不是为了自己把知识点列全。

在此声音之下，保持严肃、克制、轻学术，以完整的陈述句论证，直接给出实质内容。中文为第一语言，
术语首次出现时附英文原词。定义力求精确（形式化定义、偏序、谓词等），行文保持可读，
不堆砌术语。

> 注意：下面的「禁用清单」针对的是这些手法的**空洞 / 机器味版本**，而非原作者那种承载了
> 推理的「我们 / 读者」声音。有内容的「我们来把骨架拆开看」要保留，空转的「我们来看一下」
> 才删。判断标准始终是：这句话是否在为读者推进理解。

下列写法一律避免（它们是"机器味"与口水话的主要来源）：

- **破折号**：中文 `——` 与西文 `—` 都不用。补充、强调、转折改用逗号、冒号、括号或独立句子。
- **垫场开头**：删去"在讨论 X 之前""需要强调的是""值得注意的是""事实上""首先"这类开场垫话，
  直接进入正题。
- **强调拐杖**：不用"正是""恰恰""真正""其实"等渲染词；加粗只用于定义术语，不用于制造语气。
- **商业黑话**：不用"赋能""抓手""闭环"等。
- **副词**：尽量不用"完全""极其""非常""显然""几乎""刻意""始终""总是"等副词，改以具体陈述代替。
- **空泛断言**：不写"有着深远的影响"却不说明影响是什么；给出具体后果。
- **元叙述**：少用"本节将""下面给出""如前所述""本书的主题"这类对写作本身的旁白。
- **二元对照**：不用"不是 X，而是 Y"或成对的"越……越……"作为修辞拐杖；把每个事实直接陈述。
- **否定式罗列**：不靠堆叠"既不……也不……"来定义事物。
- **戏剧化断句**：不写短促的口号式断片；用完整句子。
- **设问铺垫**：不用"那么会怎样？"再自答；直接给出结论。
- **虚假能动**：不写"编译器可以做任何事"这类含糊归因；具体说明是谁、在什么规则下、做了什么。
- **抽离旁观的导游口吻**：少用"我们来看""我们不妨"。
- **被动语态**：优先主动语态，避免无谓的"被"字句。
- **偷懒的极端词**：除非确实成立，不用"所有""唯一""永远""完全""任何""绝大多数"。

## 九、机械规则（务必遵守）

- **保留 Hugo frontmatter**：每个 `.md` 顶部的 `weight` / `title` /
  `bookCollapseSection` 驱动站点导航，重写正文时必须原样保留，否则目录会断裂。
- 每次提交聚焦**单一范围**，提交信息说明"做了什么"与"为什么"。
